# 外部系统与集成约定

> **用途**：记录当前项目依赖的外部系统、接口、消息、文件、账号和联调方式。
> **维护方**：集成负责人 / 模块 owner / 测试负责人

## 1. 系统拓扑

```text
<当前系统>
  ├─ HTTP / RPC -> <外部系统 A>
  ├─ MQ / Event -> <外部系统 B>
  ├─ File / Object Storage -> <外部系统 C>
  └─ Webhook / Callback <- <外部系统 D>
```

## 3. API / SDK 约定

| 系统 | API/SDK | 调用方 | 鉴权 | 超时/重试 | 真实联调入口 |
|------|---------|--------|------|-----------|--------------|
| `<系统 A>` | `<路径/方法/SDK 方法>` | `<模块>` | `<方式>` | `<策略>` | `<curl/控制台/模拟器>` |

维护要求：

- 真实 API 路径、参数和响应以源码、官方文档或联调记录为准。
- 不允许凭历史经验补字段；字段语义不确定时必须回到事实源。
- 新增、删除或重命名字段时，要反查所有调用方、DTO、Mapper、日志和测试。

| Topic / Queue / Event | 生产者 | 消费者 | Payload 真源 | 幂等键 | 验证方式 |
|-----------------------|--------|--------|--------------|--------|----------|

异步任务完成证据至少包含：

- 生产侧日志或发送记录。
- 消费侧日志或处理记录。
- 状态推进、数据库落库或下游回调。
- 重试、死信或补偿策略说明。

## 7. 联调与验收要求

- 有可用沙箱或测试环境时，不能只提交 mock 结果。
- 如果真实联调暂不可用，必须写明阻塞原因、owner、恢复条件和临时替代验证。
- `test-report.md` 必须包含实际请求、实际响应、日志或数据证据。
- 对外部系统的兼容逻辑必须可观测，不能静默吞错。
