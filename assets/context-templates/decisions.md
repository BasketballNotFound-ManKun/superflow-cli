# 架构与工程决策

> **用途**：沉淀当前项目的长期架构选择、工程约定和取舍原因。
> **维护方**：架构负责人 / 技术负责人 / 模块 owner
> **推荐格式**：ADR（Architecture Decision Record）

## ADR-001: <决策标题>

- **状态**：拟定 / 已采纳 / 已废弃 / 被替代
- **日期**：YYYY-MM-DD
- **背景**：为什么需要这个决策，原问题是什么。
- **决策**：最终选择的方案。
- **替代方案**：
  - 方案 A：优点、缺点。
  - 方案 B：优点、缺点。
- **影响范围**：涉及的仓库、模块、表、接口、任务或部署方式。
- **后果**：带来的收益、成本、风险和后续约束。
- **验证方式**：如何确认决策被正确执行。

## 4. 测试与交付约定

- 单元测试验证局部逻辑，不等价于真实链路通过。
- 涉及外部入口的任务必须留下真实入口证据。
- 涉及数据库的任务必须留下表结构、数据写入或查询证据。
- 涉及异步链路的任务必须留下消息、日志、状态推进或重试证据。
- 声明完成前必须执行 Superflow guard 和项目自己的验证命令。

## 5. Superflow 使用约定

- OpenSpec/SDD 负责 WHAT 和合同。
- Superpowers 负责源码级 HOW、TDD 顺序、review 和验证纪律。
- `.sdd/state.yaml` 和 `.sdd/handoff/` 是防止上下文漂移的状态真源。
- Worker 并行开发时，必须明确各自只修改的文档和代码范围。
- 如果文档、代码或测试报告发生变化，必须刷新 handoff hash。
