```markdown
[描述当前正在解决什么问题，处于哪个开发阶段]

### 原始提示词（Prompt）
```
完整的提示词内容
```

### AI 输出摘要
[AI 返回的核心内容，2-3 句话概括]

### 采纳情况
- [x] 直接采纳（无需修改）
- [ ] 部分采纳（修改后使用）
- [ ] 未采纳（重新提问或人工解决）

### 人工修改点
如部分采纳，列出修改内容：

### 校验方式
[如何验证 AI 输出是否正确，如：编译通过 / 单元测试通过 / 人工 Review]

### 经验教训
[如果 AI 输出有问题，记录问题原因和避免方式]
```

## 完整示例

```markdown
# AI 提示词使用记录

## 条目 1

### 时间
2026-05-09 09:00

### 上下文
需要设计门禁系统的数据库表结构，包含人员、门禁设备、通行记录。

### 原始提示词
```
帮我设计一个人行门禁系统的数据库表结构，使用 MySQL。
需要包含：
1. 人员信息表（姓名、手机号、人脸特征、状态）
2. 门禁设备表（设备编号、位置、状态）
3. 通行记录表（人员、设备、时间、方式、结果）
要求：
- 使用 InnoDB 引擎
- 有合适的索引
- 字段命名用下划线
- 包含 created_at / updated_at
```

### AI 输出摘要
生成了 3 张表的 DDL，包含主键、索引、外键和注释。

### 采纳情况
- [x] 部分采纳

### 人工修改点
1. 人脸特征字段从 TEXT 改为 VARCHAR(2048)（实际 base64 长度可控）
2. 增加了 `tenant_id` 字段（多租户需求，AI 未考虑）
3. 索引调整：通行记录增加 `(device_id, pass_time)` 联合索引（按设备查历史是高频操作）

### 校验方式
- 在 MySQL 8.0 执行 DDL，确认无语法错误
- 使用 EXPLAIN 验证索引生效

### 经验教训
AI 生成表结构时容易遗漏业务特定的字段（如 tenant_id），需要在提示词中明确说明业务背景。

---

## 条目 2

### 时间
2026-05-09 11:00

### 上下文
实现设备回调接口，处理门禁设备的刷卡事件。

### 原始提示词
```
帮我写一个 Spring Boot Controller，接收门禁设备的回调请求。
请求格式：
POST /device/callback
{
  "deviceId": "string",
  "cardNo": "string",
  "timestamp": "2026-05-09T10:00:00Z",
  "eventType": "CARD_SWIPE"
}

需要：
1. 校验 deviceId 存在
2. 根据 cardNo 查询人员
3. 判断是否有权限
4. 记录通行日志
5. 返回开门指令
```

### AI 输出摘要
生成了 Controller + Service 代码，包含参数校验、业务逻辑、异常处理。

### 采纳情况
- [x] 部分采纳

### 人工修改点
1. 增加了幂等性校验（相同 cardNo + timestamp 5 分钟内不重复处理）
2. 使用分布式锁（Redisson）防止并发重复开门
3. 将事务边界从 Controller 移到 Service 层

### 校验方式
- 单元测试：Mock Service，验证参数校验逻辑
- 集成测试：启动应用，使用 curl 模拟回调，验证数据库记录

### 经验教训
AI 生成的代码缺少并发安全考虑，涉及设备控制必须考虑幂等性和并发。
```
